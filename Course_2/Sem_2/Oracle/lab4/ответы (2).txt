1. База данных Oracle состоит из нескольких файлов, таких как управляющие файлы (control files), файлы данных (data files), журналы повтора (redo logs), файлы параметров (parameter files), файлы словаря данных (data dictionary files) и другие вспомогательные файлы.

2. При инсталляции Oracle 12c создаются табличные пространства SYSTEM, SYSAUX, UNDO и TEMP.

Табличное пространство SYSTEM создается для хранения системных объектов базы данных, включая словарь данных и метаданные. Это табличное пространство также используется для хранения информации о системных пользователей и привилегиях.

Табличное пространство SYSAUX создается для хранения дополнительных системных объектов, которые не могут быть помещены в табличное пространство SYSTEM.

Табличное пространство UNDO используется для хранения информации об изменениях, внесенных в базу данных, и позволяет откатывать транзакции.

Табличное пространство TEMP используется для временного хранения данных, например, при выполнении операций сортировки и объединения данных.

3. Свойство табличных пространств smallfile/bigfile определяет максимальный размер файла данных в табличном пространстве. Если свойство установлено как smallfile, то максимальный размер файла данных будет ограничен, например, до 32 Гб в Oracle 12c. Если свойство установлено как bigfile, то максимальный размер файла данных будет значительно больше, например, до 128 Тб.

4. Свойство табличных пространств logging/nologging определяет, будут ли операции записи в табличном пространстве записываться в журнал повтора или нет. Если свойство установлено как logging, то все операции записи будут записываться в журнал повтора. Если свойство установлено как nologging, то операции записи не будут записываться в журнал повтора, что может повысить производительность, но также может увеличить риск потери данных при сбоях.

5. Свойство табличных пространств offline/online в Oracle определяет, доступно ли данное табличное пространство для изменений и запросов. Когда табличное пространство находится в режиме ONLINE, оно доступно для всех операций, включая чтение, запись, изменение и удаление данных. Когда табличное пространство находится в режиме OFFLINE, оно не доступно для запросов и изменений данных.

6. Чтобы выяснить имя UNDO-табличного пространства, используемого в Oracle 12c, можно выполнить следующий запрос SQL:

SELECT tablespace_name
FROM dba_tablespaces
WHERE contents = 'UNDO';

Этот запрос вернет имя табличного пространства, используемого для хранения UNDO-информации в базе данных.

7. Журналы повтора (redo logs) в Oracle необходимы для обеспечения целостности данных и возможности восстановления базы данных в случае сбоев. Они записывают все изменения, происходящие в базе данных, и позволяют восстановить базу данных до момента сбоя.

8. Мультиплексирование журналов повтора - это процесс создания нескольких копий журналов повтора и записи изменений во все эти копии одновременно. Это обеспечивает дополнительную защиту данных от потери в случае сбоев. Если один журнал повтора не доступен, база данных может продолжать запись изменений в другие журналы повтора.

9. Группа журналов повтора (redo log group) - это набор журналов повтора, которые используются последовательно для записи изменений в базе данных. Каждый журнал повтора имеет фиксированный размер и когда он заполняется, база данных начинает запись изменений в следующий журнал в группе. Когда все журналы в группе заполнены, они начинают запись изменений в первый журнал в группе.

10. Максимальное количество групп журналов повтора и максимальное количество файлов в группе регламентируются параметрами в конфигурационном файле базы данных (init.ora). Параметр MAXLOGFILES определяет максимальное количество групп журналов повтора, а параметр MAXLOGMEMBERS определяет максимальное количество файлов в каждой группе.

11. Параметры MAXLOGFILES и MAXLOGMEMBERS находятся в конфигурационном файле базы данных Oracle (init.ora). Чтобы просмотреть значения этих параметров, можно выполнить следующий запрос SQL:

SELECT name, value
FROM v$parameter
WHERE name IN ('MAXLOGFILES', 'MAXLOGMEMBERS');

Чтобы изменить значения этих параметров, необходимо изменить соответствующие значения в файле init.ora и перезапустить инстанс базы данных.

12. Чтобы определить группу повтора, в настоящий момент используемую инстансом Oracle 12c, можно выполнить следующий запрос SQL:

SELECT GROUP#
FROM v$log
WHERE STATUS = 'CURRENT';

Этот запрос вернет номер группы журналов повтора, которую в настоящее время использует инстанс базы данных.

13. SCN расшифровывается как системное номер изменения (system change number) в Oracle. SCN - это 64-битное целое число, которое уникально идентифицирует каждое изменение данных в базе данных.

*14. Для просмотра последовательности SCN в журналах повтора можно выполнить следующую команду SQL*Plus:

SELECT first_time, next_time, dbid, thread#, sequence#, first_change#, next_change# 
FROM v$log_history 

15. Архивы в Oracle 12c - это резервные копии журналов повтора, которые создаются и сохраняются на диске или на другом носителе для восстановления базы данных в случае сбоя.

16. Для прослеживания последовательности SCN в архивах и журналах повтора Oracle можно использовать несколько методов.

Один из таких методов - это использование команды V$ARCHIVED_LOG. Она позволяет просмотреть список всех архивных журнальных файлов, которые были созданы для базы данных, а также их связанные SCN. Для этого нужно выполнить следующую команду:

SELECT sequence#, first_time, next_time, first_change#, next_change# 
FROM v$archived_log 

В результате выполнения этой команды, вы увидите список всех архивных журнальных файлов, которые были созданы для базы данных, а также их связанные SCN.

17. Чтобы узнать, выполняется ли архивирование инстансом или нет, можно проверить значение параметра log_archive_dest_state_X, где X - это номер целевого архивного журнала. Если значение этого параметра равно ENABLE, то архивирование выполняется. Если значение DEFER, то архивирование отложено. Если значение DISABLE, то архивирование выключено.
Чтобы выяснить, выполняется ли архивирование инстансом Oracle, можно проверить значение параметра LOG_ARCHIVE_DEST_STATE_X, где X - номер целевого логического устройства архивации. Если значение параметра установлено в "ENABLE", то архивирование выполняется. Если значение установлено в "DEFER", то архивирование отложено, а если в "DISABLE", то архивирование выключено.

21. Управляющие файлы Oracle 12c (Control files) - это файлы, которые хранят метаданные базы данных, такие как структура базы данных, параметры конфигурации и информация о состоянии. Они используются для восстановления базы данных в случае сбоя.

22. Мультиплексирование управляющих файлов требуется для обеспечения отказоустойчивости. Если управляющий файл повреждается или теряется, то для восстановления базы данных необходимо иметь резервную копию управляющего файла. Если резервная копия находится на том же диске, где расположен поврежденный управляющий файл, то она может также быть повреждена. Поэтому следует использовать мультиплексирование, чтобы хранить несколько копий управляющего файла на разных дисках.

23. Количество и местоположение управляющих файлов задаются в файле конфигурации базы данных - параметр CONTROL_FILES. Этот параметр может быть задан при создании базы данных или изменен позже с помощью команды ALTER SYSTEM.

24. Для получения содержимого управляющих файлов Oracle 12c можно воспользоваться утилитой операционной системы, например, командной оболочкой (Command Prompt) в Windows.
Например: type F:\programm\Oracle\app\Admin\oradata\ORCL\CONTROL01.CTL

25. Для уменьшения или увеличения количества управляющих файлов необходимо выполнить следующие шаги:
Создать резервную копию управляющих файлов.
Остановить базу данных.
Изменить параметр CONTROL_FILES в файле конфигурации базы данных, указав новые значения для количества и местоположения управляющих файлов.
Запустить базу данных.
Проверить, что новые управляющие файлы успешно созданы и база данных работает нормально.
Удалить старые управляющие файлы, если они больше не нужны.
